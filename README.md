# 恋人手札

`恋人手札` 是一个面向情侣关系场景的双人记录与陪伴应用，计划基于 `uni-app + Vue 3 + FirstUI + uniCloud` 打造为可运行在微信小程序、H5 与 App 端的轻量产品。项目核心目标不是单纯做“笔记工具”，而是围绕情侣双方的陪伴、纪念、互动、共创和长期关系经营，沉淀出一套完整的内容记录与情感运营体系。当前小程序名称已确定为“恋人手札”，整体品牌方向会更偏温柔、陪伴感和长期关系记录。

当前仓库仍处于项目初始化阶段，已经完成了 `uni-app` 基础工程、`FirstUI` 组件接入和 `uniCloud-alipay` 云端目录预留，但业务页面、数据结构、接口能力和真实交互流程还没有正式搭建。本文档既用于说明当前项目状态，也作为后续开发规划与实施蓝图。

## 项目定位

恋人手札 的定位是“情侣双人空间 + 纪念记录 + 日常互动工具”，比普通备忘录更强调两个人之间的关系连接。产品可以围绕以下几个核心场景展开：

- 关系建立：情侣绑定、邀请码邀请、纪念日初始化、双方资料设置。
- 日常陪伴：双人日记、今日心情、聊天式碎片记录、想对你说。
- 节点纪念：恋爱天数、纪念日倒计时、生日提醒、周年策划。
- 共同成长：愿望清单、约会计划、年度目标、已完成回顾。
- 内容沉淀：图文记录、相册时间轴、视频动态、回忆卡片。
- 情感运营：节日提醒、惊喜任务、关系状态数据、温度值激励。

## 推荐的产品功能规划

下面的规划会将恋人手札从“项目壳子”推进为“可迭代上线的情侣记录产品”。

### 1. 情侣关系绑定模块

这是产品真正成立的前提，建议作为第一优先级落地。

功能目标：

- 支持用户创建情侣邀请卡。
- 支持通过邀请码、分享链接或二维码完成绑定。
- 支持查看双方昵称、头像、纪念日起始时间、恋爱天数。
- 支持解除绑定、重新绑定、关系状态变更。

建议页面：

- `pages/index/index`：改造成情侣空间首页。
- `pages/bind/create`：创建邀请页。
- `pages/bind/join`：输入邀请码加入页。
- `pages/profile/profile`：个人资料与关系管理页。

建议数据模型：

- `users`：用户基础信息。
- `couples`：情侣关系表。
- `couple_invites`：邀请记录表。

交互重点：

- 未绑定用户进入首页时，优先展示“创建情侣空间 / 输入邀请码”。
- 已绑定用户进入首页时，展示恋爱天数、伴侣信息和核心功能入口。
- 所有情侣空间数据都必须绑定 `couple_id`，避免用户数据串联错误。

### 2. 首页主工作台

首页不是简单导航页，而应该是高频使用入口和情绪氛围中心。

建议承载内容：

- 今日恋爱天数卡片。
- 最近纪念日提醒。
- 今日一句想说的话。
- 最新双人动态。
- 快捷入口：纪念日、日常记录、愿望清单、相册、约会计划。

建议首页结构：

1. 头部关系卡：情侣头像、昵称、恋爱第几天。
2. 中部提醒区：纪念日倒计时、生日、待完成约会计划。
3. 内容入口区：日常记录、相册、清单、愿望、惊喜任务。
4. 底部最近动态：最近发布的图文、日记、互动内容。

### 3. 双人日常记录模块

这是恋人手札的核心留存模块，建议尽早形成完整闭环。

功能目标：

- 支持发布文字、图片、多图日常记录。
- 支持选择心情标签、天气、地点、发布时间。
- 支持伴侣点赞、留言、回应。
- 支持按时间轴回看双人生活。

建议页面：

- `pages/feed/list`：双人动态列表。
- `pages/feed/publish`：发布图文日常。
- `pages/feed/detail`：动态详情与评论互动。

建议数据模型：

- `posts`：动态内容表。
- `post_images`：动态图片表。
- `post_comments`：评论表。
- `post_likes`：点赞表。

可扩展方向：

- 情绪趋势统计。
- 按月份生成“我们的这个月”回忆总结。
- 自动提取高频关键词生成回忆标签。

### 4. 纪念日与倒计时模块

纪念日是情侣产品最自然的高频场景之一，建议作为第二优先级主功能。

功能目标：

- 支持创建恋爱纪念日、生日、第一次见面、第一次旅行等重要节点。
- 支持展示距离下一个纪念日还有多少天。
- 支持提醒规则设置，如提前 7 天、3 天、1 天提醒。
- 支持为纪念日添加照片、文案和回忆描述。

建议页面：

- `pages/anniversary/list`
- `pages/anniversary/create`
- `pages/anniversary/detail`

建议数据模型：

- `anniversaries`：纪念日主体表。
- `anniversary_reminders`：提醒规则表。

延展能力：

- 自动生成纪念海报。
- 纪念日当天推送“回忆卡片”。
- 与约会计划联动，自动生成庆祝待办。

### 5. 愿望清单与约会计划模块

这部分负责把“记录过去”延伸到“共同规划未来”。

功能目标：

- 支持创建共同愿望清单，如旅行、拍照、礼物、打卡地。
- 支持发起约会提案，并设置时间、预算、地点、备注。
- 支持完成打卡、图片补充、心得记录。
- 支持按“想做 / 进行中 / 已完成”管理状态。

建议页面：

- `pages/wishlist/list`
- `pages/wishlist/create`
- `pages/date-plan/list`
- `pages/date-plan/detail`

建议数据模型：

- `wishlists`
- `wishlist_items`
- `date_plans`
- `date_plan_logs`

产品价值：

- 增强情侣双方共同参与感。
- 让应用不仅记录回忆，也主动制造回忆。

### 6. 情侣相册与时间轴模块

这是内容沉淀能力的升级版，适合在图文动态稳定后接入。

功能目标：

- 按时间轴展示双方共同上传的照片与视频。
- 支持按月份、纪念日、地点自动归档。
- 支持精选回忆卡片展示。
- 支持封面照片与情侣空间背景图设置。

建议页面：

- `pages/album/list`
- `pages/album/detail`
- `pages/memory/timeline`

建议数据模型：

- `albums`
- `media_assets`
- `memory_cards`

### 7. 我的与设置模块

该模块用于汇总账号信息、偏好设置、业务状态和辅助入口。

建议包含：

- 用户资料编辑。
- 伴侣信息查看。
- 关系设置与解绑管理。
- 消息通知设置。
- 关于我们、隐私协议、用户协议。
- 意见反馈与版本说明。

建议页面：

- `pages/profile/profile`
- `pages/profile/settings`
- `pages/profile/about`

## 建议的数据结构分层

为了后续扩展稳定，建议尽早统一数据分层：

- 账号层：用户身份、登录信息、基础资料。
- 关系层：情侣绑定关系、邀请记录、关系状态。
- 内容层：动态、评论、点赞、图片、相册、回忆卡片。
- 计划层：愿望清单、约会计划、任务打卡。
- 时间层：纪念日、提醒、节日计划。
- 配置层：通知设置、隐私开关、平台配置。

这样做的好处是后续无论增加统计面板、AI 回忆总结、消息提醒还是多端同步，都能在现有结构上扩展，不需要反复推翻重建。

## 推荐页面演进方案

建议不要一开始就铺太多页面，而是围绕主链路逐步搭建。

### 第一阶段：先跑通最小可用版本

目标是让用户能够完成“进入应用 -> 绑定情侣 -> 创建纪念日 -> 发布第一条双人日常”。

建议优先开发：

1. 首页改版为情侣空间首页。
2. 情侣绑定流程。
3. 纪念日创建与列表。
4. 双人日常发布与列表。
5. 我的页面基础资料与设置。

### 第二阶段：增强互动与留存

目标是提高双方打开频次和内容沉淀率。

建议增加：

1. 动态评论与点赞。
2. 愿望清单。
3. 约会计划。
4. 提醒通知。
5. 月度回忆回顾。

### 第三阶段：增强产品特色

目标是形成差异化体验，而不是停留在普通记录工具。

建议探索：

1. 回忆时间轴。
2. 节日惊喜任务。
3. 自动生成情侣海报。
4. 情绪趋势和陪伴数据统计。
5. AI 辅助文案、纪念日祝福、回忆总结。

3. 完成纪念日模块，建立时间型内容基础。
4. 完成双人日常发布与列表，建立内容型主链路。
5. 补齐“我的”页面和基础配置。
6. 增加愿望清单、约会计划、相册等增强模块。

## UI 与交互方向建议

恋人手札的视觉风格不建议走通用后台式设计，而应更偏向温暖、轻盈、陪伴感明确的情绪化产品体验。

设计建议：

- 以柔和暖色为主，如奶白、米杏、浅粉、雾橙、淡红棕。
- 首页重视氛围卡片，而不是纯列表堆叠。
- 强化“恋爱天数”“下一次纪念日”“今日互动”这类情绪锚点。
- 重要操作保持简单直接，减少过度层级跳转。
- 图文内容优先采用卡片流或时间轴形式，增强回忆感。

## 后端与云函数规划建议

虽然当前云端能力尚未开始接入，但建议尽量按真实业务标准设计。

推荐优先准备：

- 用户登录与身份校验。
- 情侣关系绑定与解绑云函数。
- 动态发布、评论、点赞接口。
- 纪念日增删改查接口。
- 文件上传能力，用于相册和动态图片。
- 消息提醒与定时任务能力。

如果后续继续使用 `uniCloud`，建议同步补充：

- 数据权限规则。
- 统一返回结构。
- 云函数错误码定义。
- 上传资源命名规范。
